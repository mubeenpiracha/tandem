# Deployment Guide

## Overview

Tandem runs as two services on Railway:
1. **Backend** — Node.js API + Slack bot + BullMQ worker
2. **Frontend** — React app (served via Railway's static hosting)

PostgreSQL and Redis are added as Railway plugins — no separate hosting needed.

---

## Environments

| Environment | Purpose | URL | Domain needed? |
|-------------|---------|-----|----------------|
| Local | Development on your machine | `localhost:3000` | No — use ngrok for Slack webhooks |
| Staging | Pre-production testing on Railway | Auto-generated by Railway (e.g. `tandem-backend-staging.up.railway.app`) | No |
| Production | Live users | Auto-generated Railway URL or custom domain | Only if you want a branded URL |

**You do not need a custom domain to launch.** Railway gives every service a public URL automatically. Add a custom domain later when you're ready to go public.

---

## Two Deployed Environments on Railway

You have one Railway account. Create two separate Railway projects:

| Project name | Purpose |
|---|---|
| `tandem-staging` | Where you test before going live |
| `tandem-production` | Where real users go |

Each project gets its own PostgreSQL, Redis, backend, and frontend. They share nothing — not the database, not the Slack app, not the environment variables.

## Staging Setup (Do This First)

### 1. Create the Railway project
- Railway dashboard → New Project → "tandem-staging"
- Add service: Deploy from GitHub repo → select your Tandem repo → backend package
- Add service: Deploy from GitHub repo → same repo → frontend package
- Add plugin: PostgreSQL
- Add plugin: Redis

Railway will give you a public URL for each service automatically. Copy the backend URL — you'll need it for Slack.

### 2. Create a Slack App for staging
- Go to [api.slack.com/apps](https://api.slack.com/apps) → Create New App → "Tandem Dev"
- Configure Event Subscriptions URL: `https://<railway-backend-staging-url>/slack/events`
- Configure Interactivity URL: `https://<railway-backend-staging-url>/slack/interactions`
- Install to your test Slack workspace
- Copy credentials to Railway staging environment variables (see below)

### 3. Set environment variables in Railway staging
In Railway → tandem-staging → backend service → Variables:

```
NODE_ENV=staging
DATABASE_URL=<auto-filled by Railway PostgreSQL plugin>
REDIS_URL=<auto-filled by Railway Redis plugin>
JWT_SECRET=<run: openssl rand -hex 64>
ENCRYPTION_KEY_CURRENT=<run: openssl rand -hex 32>
SLACK_CLIENT_ID=<from Tandem Dev Slack App>
SLACK_CLIENT_SECRET=<from Tandem Dev Slack App>
SLACK_SIGNING_SECRET=<from Tandem Dev Slack App>
GOOGLE_CLIENT_ID=<from Google Cloud project>
GOOGLE_CLIENT_SECRET=<from Google Cloud project>
GOOGLE_REDIRECT_URI=https://<railway-backend-staging-url>/api/connect/google/callback
OPENAI_API_KEY=<your key>
FRONTEND_URL=https://<railway-frontend-staging-url>
```

### 4. Run migrations
Railway runs migrations automatically on deploy if configured in the start command:
```
pnpm db:migrate && pnpm start
```

Or run manually via Railway shell:
```bash
cd packages/backend && pnpm prisma migrate deploy
```

### 5. Verify staging works
- `/api/health` returns 200
- You can log in
- Slack events arrive (install the "Tandem Dev" app to a test workspace and @mention someone)

## Going to Production

When staging is working and you're ready for real users, run the `/go-to-production` skill inside Claude Code. It will walk you through every step.

```
/go-to-production
```

Do not attempt production setup manually — the skill ensures nothing is missed and the two environments remain isolated.

## Deployment Process (Day-to-Day)

### Staging (automatic)
Every merge to `main` triggers a Railway redeploy of staging automatically via GitHub integration. No manual action needed.

### Production (manual promote)
1. Verify staging is working correctly
2. In Railway → tandem-production → Deployments → trigger a new deploy
3. Monitor logs for errors

---

## Rolling Back

**Application rollback:**
Railway keeps deployment history. In Railway → your service → Deployments → click a previous deploy → Redeploy.

**Database rollback:**
- Railway PostgreSQL has point-in-time recovery
- Take a manual backup snapshot before any major migration: Railway → PostgreSQL plugin → Backups → Create backup
- Restore from Railway dashboard if needed

**Prevention:** Never combine a destructive database change (drop column, drop table) with a code change in the same deploy. Split them: deploy the migration first, verify, then deploy the code change.

## CI/CD Pipeline (GitHub Actions)

Three workflows in `.github/workflows/`:

**`ci.yml` — on every pull request:**
- Type check (`pnpm type-check`)
- Lint (`pnpm lint`)
- Tests (`pnpm test`)
- PR cannot be merged if any fails

**`deploy-staging.yml` — on merge to `main`:**
- Railway auto-redeploys via GitHub integration (no workflow needed if Railway is connected to the repo)

**`deploy-production.yml` — manual trigger only:**
- Triggered from GitHub Actions UI
- Runs full test suite
- Deploys to Railway production project

## Health Monitoring

- Health check: `GET /api/health` → 200 with DB + Redis + Slack connection status
- Set up free uptime monitoring: [Better Uptime](https://betteruptime.com) or [UptimeRobot](https://uptimerobot.com)
  - Point at: `https://<railway-backend-production-url>/api/health`
  - Alert via email when it goes down

## Secrets Management

- Never commit secrets to git — use Railway's Variables UI
- Each environment has its own `ENCRYPTION_KEY_CURRENT` — if staging's key leaks, production is safe
- To rotate a key: add new key as `ENCRYPTION_KEY_CURRENT`, move old key to `ENCRYPTION_KEY_PREVIOUS` — tokens are re-encrypted lazily on next read
- Slack and Google tokens can be revoked from their dashboards if compromised
